미들웨어 - 서로 다른 application이 통신하는데 사용되는 SW
	ex) WAS = 클라이언트 - DB
형상관리 - SCM : Software Configuration Management
	소프트웨어의 개발과정에서 변경 사항을 관리하기 위해 개발된 인련의 활동
	ex) Git, SVN(subvision), CVN
=======
DTO(Data Transfer Object) or VO(Value Object)
	공통 코드 조회를 위해 필요한 변수들을 정의하여 묶어두고
	계층 간 데이터 교환을 위해 사용

DAO(Data Access Object) : sql문을 구현한 xml파일에서 
		해당 id값을 호출하여 데이터를 조회하거나 조작

1. DTO/VO구현
2. xml(sql문) 구현
3. DAO 구현
4. Service 구현
5. Controller 구현
6. 화면(view) 구현

MVC : Model View Controller
	웹 애플리케이션을 비즈니스 로직, 프레젠테이션 로직, 데이터로 분리하는 디자인 패턴
	1. 비즈니스 로직 : 애플리케이션의 데이터
	2. 프레젠테이션 : 애플리케이션이 사용자에게 어떻게 표시되는지
	3. 요청 처리 데이터 : 비즈니스 로직과 프레젠테이션 파트를 함께 묶은 것
	Model : 애플리케이션의 데이터와 비즈니스 로직을 담는 객체
	View : 사용자에게 모델의 정보(데이터)를 보여주는 역할
		비즈니스 로직을 포함하지 않음
		하나의 모델을 다양한 뷰에서 사용
	Controller : 모델과 뷰 사이에 발생하는 동작을 조정하는 역할
		웹으로부터 받은 요청에 가장 적합한 모델을 생선하는 것을 처리하는 역할과
		사용자에게 응답하는 적절한 뷰를 선택하여 해당 모델을 전달하는 역할
모델1 : jsp로만 구현한 웹 애플리케이션
	웹브라우저의 요청을 jsp페이지가 받아서 처리하는 구조
모델2 : 클라이언트의 요청 처리, 응답 처리, 비즈니스 로직 처리 부분을 모듈화한 구조

테스트케이스
	식별자 : 항목 식별자, 일련변호
	테스트 항목 : 테스트 대상(모듈, ....)
	입력 명세 : 입력 데이터 또는 테스트 조건
	출력 명세 : 테스트 케이스 수행 시 예상되는 출력 결과
	환경 설정 : 필요한 하드웨어나 소프트웨어의 환경
	특수 절차 요구 : 테스트 케이스 수행 시 특별히 요구되는 절차
	의존성 기술 : 테스트 케이스 간의 의존성
===============
로그(log) : 프로그램 개발이나 운영 시 발생하는 [문제점]을 [추적]하거나
	운영 상태를 [모니터링] 하기 위한 텍스트(기록)
	error : 요청 처리 중 일반적 에러 발생한 상태
		프로그램 동작에 큰 문제가 발생 >> 즉시 문제를 조사해야한다는 것
	warn : 프로그램의 실행에는 문제가 없으나 향후 에러의 원인이 될 수 있다 (잠재적)
	info : 로그인, 상태 변경과 같은 정보성 메세지
	debug : 개발 시 디버깅 용도로 사용할 메세지

제품 소프트웨어 패키징
	소프트웨어 패키징 : 배포용 설치 파일을 만드는 것
	사용자 중심 패키지 수행
패키징 시 고려사항
	사용자의 시스템 환경(CPU, OS(운영체제), 메모리 등에 필요한 최소 환경을 정의)
	UI(사용자 환경)는 사용자가 눈으로 직접 확인할 수 있도록
		시각적인 자료와 함께 제공하고 메뉴얼과 일치시켜 패키징
	소프트웨어는 단숨히 패키징하여 배포하는 것으로 끝나는 것이 아니라
		하드웨어와 함께 관리될 수 있도록 Manage Service 형태로 제공하는 것이 좋음
	사용자에게 배포되는 소프트웨어이므로 내부 컨텐츠에 대해 암호화 및 보안을 고려
	다른 여러 컨텐츠 및 단말기 간 DRM(디지털 저작권 관리) 연동을 고려
	제품 소프트웨어 종류에 적합한 암호화 알고리즘을 적용
